# Ralphinho RFCパイプライン

[humanplane](https://github.com/humanplane)スタイルのRFC分解パターンとマルチユニットオーケストレーションワークフローに着想を得ています。

機能が単一のエージェントパスには大きすぎ、独立して検証可能なワークユニットに分割する必要がある場合にこのスキルを使用します。

## パイプラインステージ

1. RFC受付
2. DAG分解
3. ユニット割り当て
4. ユニット実装
5. ユニット検証
6. マージキューと統合
7. 最終システム検証

## ユニット仕様テンプレート

各ワークユニットには以下を含める必要があります:
- `id`
- `depends_on`
- `scope`
- `acceptance_tests`
- `risk_level`
- `rollback_plan`

## 複雑度ティア

- ティア1: 単一ファイルの編集、決定論的なテスト
- ティア2: 複数ファイルの動作変更、中程度のインテグレーションリスク
- ティア3: スキーマ/認証/パフォーマンス/セキュリティの変更

## ユニットごとの品質パイプライン

1. リサーチ
2. 実装計画
3. 実装
4. テスト
5. レビュー
6. マージ準備完了レポート

## マージキュールール

- 未解決の依存関係失敗があるユニットをマージしないこと。
- ユニットブランチを常に最新のインテグレーションブランチにリベースすること。
- キューされた各マージの後にインテグレーションテストを再実行すること。

## リカバリー

ユニットが停滞した場合:
- アクティブキューから退出
- 調査結果をスナップショット
- 狭められたユニットスコープを再生成
- 更新された制約でリトライ

## 出力

- RFC実行ログ
- ユニットスコアカード
- 依存関係グラフスナップショット
- インテグレーションリスクサマリー
