# テストカバレッジ

テストカバレッジを分析し、ギャップを特定し、80%以上のカバレッジに到達するための不足テストを生成します。

## ステップ1：テストフレームワークの検出

| インジケーター | カバレッジコマンド |
|-----------|-----------------|
| `jest.config.*`またはpackage.jsonのjest | `npx jest --coverage --coverageReporters=json-summary` |
| `vitest.config.*` | `npx vitest run --coverage` |
| `pytest.ini` / `pyproject.toml` pytest | `pytest --cov=src --cov-report=json` |
| `Cargo.toml` | `cargo llvm-cov --json` |
| JaCoCo付き`pom.xml` | `mvn test jacoco:report` |
| `go.mod` | `go test -coverprofile=coverage.out ./...` |

## ステップ2：カバレッジレポートの分析

1. カバレッジコマンドを実行
2. 出力を解析（JSONサマリーまたはターミナル出力）
3. **80%未満のカバレッジ**のファイルを最も低いものから順にリスト
4. カバレッジ不足の各ファイルについて特定：
   - 未テストの関数またはメソッド
   - 不足しているブランチカバレッジ（if/else、switch、エラーパス）
   - 分母を膨らませるデッドコード

## ステップ3：不足テストの生成

カバレッジ不足の各ファイルに対して、以下の優先順位でテストを生成します：

1. **ハッピーパス** — 有効な入力によるコア機能
2. **エラーハンドリング** — 無効な入力、データ欠落、ネットワーク障害
3. **エッジケース** — 空の配列、null/undefined、境界値（0、-1、MAX_INT）
4. **ブランチカバレッジ** — 各if/else、switchケース、三項演算子

### テスト生成ルール

- テストはソースの隣に配置：`foo.ts` → `foo.test.ts`（またはプロジェクトの規約に従う）
- プロジェクトの既存テストパターンを使用（インポートスタイル、アサーションライブラリ、モッキングアプローチ）
- 外部依存関係をモック（データベース、API、ファイルシステム）
- 各テストは独立 — テスト間で共有のミュータブル状態を持たない
- テストに説明的な名前を付ける：`test_create_user_with_duplicate_email_returns_409`

## ステップ4：検証

1. テストスイート全体を実行 — すべてのテストが通過すること
2. カバレッジを再実行 — 改善を確認
3. まだ80%未満の場合、残りのギャップに対してステップ3を繰り返す

## ステップ5：レポート

前後の比較を表示します：

```
Coverage Report
──────────────────────────────
File                   Before  After
src/services/auth.ts   45%     88%
src/utils/validation.ts 32%    82%
──────────────────────────────
Overall:               67%     84%  ✅
```

## 重点領域

- 複雑な分岐を持つ関数（高い循環的複雑度）
- エラーハンドラーとcatchブロック
- コードベース全体で使用されるユーティリティ関数
- APIエンドポイントハンドラー（リクエスト→レスポンスフロー）
- エッジケース：null、undefined、空文字列、空配列、ゼロ、負数
