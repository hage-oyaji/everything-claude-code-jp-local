# リファクタリング＆クリーンアップ

各ステップでテスト検証を行いながら、安全にデッドコードを特定・削除します。

## ステップ1：デッドコードの検出

プロジェクトタイプに基づいて解析ツールを実行します：

| ツール | 検出内容 | コマンド |
|------|--------------|---------|
| knip | 未使用のエクスポート、ファイル、依存関係 | `npx knip` |
| depcheck | 未使用のnpm依存関係 | `npx depcheck` |
| ts-prune | 未使用のTypeScriptエクスポート | `npx ts-prune` |
| vulture | 未使用のPythonコード | `vulture src/` |
| deadcode | 未使用のGoコード | `deadcode ./...` |
| cargo-udeps | 未使用のRust依存関係 | `cargo +nightly udeps` |

ツールが利用できない場合は、Grepを使用してインポートされていないエクスポートを検出します：
```
# Find exports, then check if they're imported anywhere
```

## ステップ2：検出結果の分類

検出結果を安全性ティアに分類します：

| ティア | 例 | アクション |
|------|----------|--------|
| **SAFE** | 未使用のユーティリティ、テストヘルパー、内部関数 | 安心して削除 |
| **CAUTION** | コンポーネント、APIルート、ミドルウェア | 動的インポートや外部コンシューマーがないことを確認 |
| **DANGER** | 設定ファイル、エントリーポイント、型定義 | 変更前に調査 |

## ステップ3：安全な削除ループ

各SAFEアイテムに対して：

1. **テストスイート全体を実行** — ベースラインを確立（すべてグリーン）
2. **デッドコードを削除** — Editツールで外科的に削除
3. **テストスイートを再実行** — 何も壊れていないことを確認
4. **テストが失敗した場合** — `git checkout -- <file>`で即座に元に戻し、このアイテムをスキップ
5. **テストが通った場合** — 次のアイテムへ

## ステップ4：CAUTIONアイテムの処理

CAUTIONアイテムを削除する前に：
- 動的インポートを検索：`import()`、`require()`、`__import__`
- 文字列参照を検索：設定ファイル内のルート名、コンポーネント名
- パブリックパッケージAPIからエクスポートされているか確認
- 外部コンシューマーがないことを確認（公開パッケージの場合は依存先を確認）

## ステップ5：重複の統合

デッドコード削除後、以下を確認します：
- 類似関数（80%以上類似）— 1つに統合
- 冗長な型定義 — 統合
- 価値を追加しないラッパー関数 — インライン化
- 目的のない再エクスポート — 間接参照を削除

## ステップ6：サマリー

結果を報告します：

```
Dead Code Cleanup
──────────────────────────────
Deleted:   12 unused functions
           3 unused files
           5 unused dependencies
Skipped:   2 items (tests failed)
Saved:     ~450 lines removed
──────────────────────────────
All tests passing ✅
```

## ルール

- **テストを実行せずに削除しない**
- **一度に1つの削除** — アトミックな変更でロールバックを容易にする
- **不確かな場合はスキップ** — 本番環境を壊すよりデッドコードを残す方がよい
- **クリーンアップ中にリファクタリングしない** — 関心事の分離（先にクリーンアップ、後でリファクタリング）
