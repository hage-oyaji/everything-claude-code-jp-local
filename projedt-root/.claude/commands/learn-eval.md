# /learn-eval - 抽出、評価、そして保存

`/learn` に品質ゲート、保存場所の決定、ナレッジ配置の認識を追加したコマンドです。スキルファイルを書き込む前にこれらを実行します。

## 抽出対象

以下を探します：

1. **エラー解決パターン** — 根本原因 + 修正 + 再利用性
2. **デバッグテクニック** — 非自明なステップ、ツールの組み合わせ
3. **ワークアラウンド** — ライブラリの癖、APIの制限、バージョン固有の修正
4. **プロジェクト固有のパターン** — 規約、アーキテクチャの決定、統合パターン

## プロセス

1. セッションを確認し、抽出可能なパターンを探す
2. 最も価値のある/再利用可能なインサイトを特定

3. **保存場所を決定：**
   - 質問: 「このパターンは別のプロジェクトでも役立つか？」
   - **グローバル** (`~/.claude/skills/learned/`): 2つ以上のプロジェクトで使える汎用パターン（bash互換性、LLM APIの動作、デバッグテクニックなど）
   - **プロジェクト** (現在のプロジェクトの `.claude/skills/learned/`): プロジェクト固有の知識（特定の設定ファイルの癖、プロジェクト固有のアーキテクチャ決定など）
   - 迷った場合はグローバルを選択（グローバル→プロジェクトへの移動はその逆より容易）

4. 以下のフォーマットでスキルファイルをドラフト：

```markdown
---
name: pattern-name
description: "Under 130 characters"
user-invocable: false
origin: auto-extracted
---

# [Descriptive Pattern Name]

**Extracted:** [Date]
**Context:** [Brief description of when this applies]

## Problem
[What problem this solves - be specific]

## Solution
[The pattern/technique/workaround - with code examples]

## When to Use
[Trigger conditions]
```

5. **品質ゲート — チェックリスト + 総合判定**

   ### 5a. 必須チェックリスト（実際にファイルを読んで検証）

   ドラフトを評価する前に以下の**すべて**を実行：

   - [ ] `~/.claude/skills/` および関連プロジェクトの `.claude/skills/` ファイルをキーワードでgrepし、コンテンツの重複をチェック
   - [ ] MEMORY.md（プロジェクトとグローバルの両方）の重複をチェック
   - [ ] 既存のスキルへの追記で十分かどうかを検討
   - [ ] これが再利用可能なパターンであり、一回限りの修正ではないことを確認

   ### 5b. 総合判定

   チェックリストの結果とドラフトの品質を総合的に評価し、以下から**1つ**を選択：

   | 判定 | 意味 | 次のアクション |
   |---------|---------|-------------|
   | **Save** | ユニーク、具体的、適切なスコープ | ステップ6へ進む |
   | **Improve then Save** | 価値はあるが改善が必要 | 改善点をリスト→修正→再評価（1回） |
   | **Absorb into [X]** | 既存のスキルに追記すべき | ターゲットスキルと追加内容を表示→ステップ6 |
   | **Drop** | 些末、冗長、または抽象的すぎる | 理由を説明して終了 |

   **ガイドライン次元**（判定に寄与するが、スコアリングはしない）：

   - **具体性と実行可能性**: すぐに使えるコード例やコマンドが含まれている
   - **スコープの適合性**: 名前、トリガー条件、コンテンツが一貫しており、単一のパターンに集中している
   - **ユニーク性**: 既存のスキルでカバーされていない価値を提供（チェックリスト結果に基づく）
   - **再利用性**: 将来のセッションで現実的なトリガーシナリオが存在する

6. **判定別の確認フロー**

   - **Improve then Save**: 必要な改善点 + 修正済みドラフト + 更新されたチェックリスト/判定を提示。修正後の判定が**Save**であればユーザー確認後に保存、そうでなければ新しい判定に従う
   - **Save**: 保存パス + チェックリスト結果 + 1行の判定理由 + 完全なドラフトを提示→ユーザー確認後に保存
   - **Absorb into [X]**: ターゲットパス + 追加内容（diff形式） + チェックリスト結果 + 判定理由を提示→ユーザー確認後に追記
   - **Drop**: チェックリスト結果 + 理由のみを表示（確認不要）

7. 決定された場所に保存/追記

## ステップ5の出力フォーマット

```
### Checklist
- [x] skills/ grep: no overlap (or: overlap found → details)
- [x] MEMORY.md: no overlap (or: overlap found → details)
- [x] Existing skill append: new file appropriate (or: should append to [X])
- [x] Reusability: confirmed (or: one-off → Drop)

### Verdict: Save / Improve then Save / Absorb into [X] / Drop

**Rationale:** (1-2 sentences explaining the verdict)
```

## 設計上の理由

このバージョンでは、以前の5次元数値スコアリング方式（具体性、実行可能性、スコープ適合性、非冗長性、カバレッジを1-5でスコアリング）をチェックリストベースの総合判定システムに置き換えました。最新のフロンティアモデル（Opus 4.6+）は優れた文脈判断能力を持っています。豊かな定性的シグナルを数値スコアに無理に変換すると、ニュアンスが失われ、誤解を招く合計値になりかねません。総合的なアプローチにより、モデルがすべての要素を自然に重み付けでき、より正確な保存/破棄の判断が可能になります。明示的なチェックリストにより、重要なチェックが漏れることもありません。

## 注意事項

- 些末な修正（タイポ、単純な構文エラー）は抽出しない
- 一回限りの問題（特定のAPI障害など）は抽出しない
- 将来のセッションで時間を節約するパターンに焦点を当てる
- スキルは焦点を絞る — 1スキル1パターン
- 判定がAbsorbの場合、新しいファイルを作成せず既存のスキルに追記する
