# Aside コマンド

タスクの途中で質問し、即座に的確な回答を得て、中断した箇所からそのまま作業を続行できます。現在のタスク、ファイル、コンテキストは一切変更されません。

## 使用場面

- Claudeが作業中に気になることがあり、勢いを失いたくないとき
- Claudeが編集中のコードについて簡単な説明が欲しいとき
- タスクを脱線させずに、判断について別の意見や確認が欲しいとき
- Claudeが作業を進める前に、エラー、概念、パターンを理解したいとき
- 現在のタスクとは無関係なことを、新しいセッションを開始せずに聞きたいとき

## 使い方

```
/aside <質問>
/aside what does this function actually return?
/aside is this pattern thread-safe?
/aside why are we using X instead of Y here?
/aside what's the difference between foo() and bar()?
/aside should we be worried about the N+1 query we just added?
```

## プロセス

### ステップ1: 現在のタスク状態を凍結する

回答の前に、以下をメンタルに記録する:
- アクティブなタスクは何か？（どのファイル、機能、または問題に取り組んでいたか）
- `/aside` が呼び出された時点で、どのステップが進行中だったか？
- 次に何が行われる予定だったか？

aside中は、ファイルの編集、作成、削除を一切行ってはならない。

### ステップ2: 質問に直接回答する

完全かつ有用でありながら、最も簡潔な形で質問に回答する。

- 推論ではなく、回答を先に述べる
- 短く保つ — 詳細な説明が必要な場合は、タスク完了後に深掘りすることを提案する
- 現在のファイルまたは作業中のコードに関する質問の場合、正確に参照する（関連する場合はファイルパスと行番号を含む）
- 回答にファイルの読み取りが必要な場合は読み取る — ただし読み取りのみで、書き込みは行わない

回答のフォーマット:

```
ASIDE: [質問を簡潔に言い換え]

[回答をここに記述]

— タスクに戻る: [行っていた作業の一行説明]
```

### ステップ3: メインタスクを再開する

回答を提供した後、一時停止した正確なポイントからアクティブなタスクを即座に続行する。asideの回答がブロッカーや現在のアプローチを再考する理由を明らかにした場合を除き、再開の許可を求めない（エッジケースを参照）。

## エッジケース

**質問が未提供の場合（`/aside` の後に何もない場合）:**
以下のように応答する:
```
ASIDE: 質問が提供されていません

何について知りたいですか？（質問してください。現在のタスクコンテキストを失わずに回答します）

— タスクに戻る: [行っていた作業の一行説明]
```

**質問が現在のタスクの潜在的な問題を明らかにした場合:**
再開前に明確にフラグを立てる:
```
ASIDE: [回答]

⚠️ 注意: この回答は現在のアプローチに[問題]があることを示唆しています。続行する前にこれに対処しますか？それとも計画通りに進めますか？
```
ユーザーの判断を待ってから再開する。

**質問が実際にはタスクのリダイレクトである場合（サイドクエスチョンではない場合）:**
質問が構築内容の変更を示唆している場合（例: `/aside actually, let's use Redis instead`）、確認する:
```
ASIDE: それはサイドクエスチョンではなく、方向転換のようです。
次のどちらを希望しますか？
  (a) 情報としてのみ回答し、現在の計画を維持する
  (b) 現在のタスクを一時停止してアプローチを変更する
```
ユーザーの回答を待つ — 推測で判断しない。

**質問が現在開いているファイルまたはコードについての場合:**
ライブコンテキストから回答する。ファイルがセッション中に以前読み取られている場合は、直接参照する。読み取られていない場合は、今読み取り（読み取りのみ）、ファイル:行番号の参照と共に回答する。

**アクティブなタスクがない場合（`/aside` 呼び出し時に何も進行中でない場合）:**
応答の形式を統一するために、標準のラッパーを使用する:
```
ASIDE: [質問を簡潔に言い換え]

[回答をここに記述]

— タスクに戻る: 再開するアクティブなタスクはありません
```

**質問に長い回答が必要な場合:**
簡潔に要点を回答し、その後提案する:
```
短い版は以上です。[現在のタスク]を終えた後に、より詳しい説明が必要ですか？
```

**連続する複数の `/aside` 質問:**
順番にそれぞれ回答する。最後の回答の後、メインタスクを再開する。一連のaside間でタスクの状態を失わないこと。

**asideの回答がコード変更の必要性を示唆した場合:**
必要な変更を記録するが、aside中には実行しない:
```
ASIDE: [回答]

📝 修正推奨: [何を変更すべきか]。現在のタスク完了後にフラグを立てます。今すぐ対処したい場合はお知らせください。
```

**質問が曖昧またはあいまいすぎる場合:**
回答に必要な情報を得るための最短の質問を1つだけ聞く。複数の質問をしない。

## 出力例

```
User: /aside what does fetchWithRetry() actually do?

ASIDE: fetchWithRetry()は何をするか？

fetchWithRetry() (src/api/retry.ts:12) はリクエストを最大3回まで
指数バックオフ（250ms → 500ms → 1s）で再試行します。5xxレスポンスと
ネットワークエラーの場合のみ再試行し、4xxエラーは最終的なものとして再試行しません。

— タスクに戻る: src/middleware/auth.tsの認証ミドルウェアのリファクタリング
```

```
User: /aside is the approach we're taking thread-safe?

ASIDE: 現在のアプローチはスレッドセーフか？

いいえ — src/cache/store.ts:34の共有キャッシュオブジェクトはロックなしで
変更されています。並行リクエスト下ではこれは競合状態です。シングルプロセスの
Node.jsサーバーではリスクは低いですが、ワーカースレッドやクラスタリングでは
実際の問題になります。

⚠️ 注意: これは構築中の機能に影響する可能性があります。今すぐ対処しますか？それとも続行してフォローアップで修正しますか？
```

## 注意事項

- aside中にファイルを変更しないこと — 読み取りのみ
- asideは会話の一時停止であり、新しいタスクではない — 元のタスクは必ず再開すること
- 回答は簡潔に: 目標はユーザーを素早くアンブロックすることであり、講義を提供することではない
- asideがより大きな議論を引き起こした場合、asideがブロッカーを明らかにしない限り、現在のタスクを先に終わらせる
- asideはタスクの結果に明らかに関連しない限り、セッションファイルに保存されない
