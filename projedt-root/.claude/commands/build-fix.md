# ビルドと修正

最小限で安全な変更により、ビルドエラーと型エラーを段階的に修正します。

## ステップ1: ビルドシステムの検出

プロジェクトのビルドツールを特定し、ビルドを実行する:

| 判定基準 | ビルドコマンド |
|-----------|---------------|
| `package.json` に `build` スクリプトあり | `npm run build` または `pnpm build` |
| `tsconfig.json`（TypeScriptのみ） | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall -q .` または `mypy .` |

## ステップ2: エラーの解析とグループ化

1. ビルドコマンドを実行し、stderrをキャプチャする
2. ファイルパスごとにエラーをグループ化する
3. 依存関係の順序でソートする（ロジックエラーより先にインポート/型を修正）
4. 進捗追跡のためにエラー総数をカウントする

## ステップ3: 修正ループ（1つずつエラーを修正）

各エラーに対して:

1. **ファイルを読み取る** — Readツールでエラー周辺のコンテキスト（エラー前後10行）を確認する
2. **診断する** — 根本原因を特定する（インポート不足、型の不一致、構文エラー）
3. **最小限に修正する** — エラーを解決する最小限の変更にEditツールを使用する
4. **ビルドを再実行する** — エラーが解消され、新たなエラーが発生していないことを確認する
5. **次へ進む** — 残りのエラーの処理を続行する

## ステップ4: ガードレール

以下の場合はユーザーに確認する:
- 修正が**解決するよりも多くのエラーを引き起こす**場合
- **同じエラーが3回試行しても解消しない**場合（より深い問題の可能性）
- 修正に**アーキテクチャの変更**が必要な場合（単なるビルド修正ではない）
- ビルドエラーが**依存関係の不足**に起因する場合（`npm install`、`cargo add` 等が必要）

## ステップ5: サマリー

結果を表示する:
- 修正済みエラー（ファイルパス付き）
- 残存エラー（ある場合）
- 新たに発生したエラー（ゼロであるべき）
- 未解決の問題に対する次のステップの提案

## リカバリー戦略

| 状況 | 対応 |
|-----------|--------|
| モジュール/インポートが見つからない | パッケージがインストール済みか確認し、インストールコマンドを提案する |
| 型の不一致 | 両方の型定義を確認し、より狭い型を修正する |
| 循環依存 | インポートグラフで循環を特定し、分離を提案する |
| バージョン競合 | `package.json` / `Cargo.toml` のバージョン制約を確認する |
| ビルドツールの設定ミス | 設定ファイルを確認し、動作するデフォルトと比較する |

安全のため、一度に1つのエラーを修正する。リファクタリングよりも最小限の差分を優先する。
