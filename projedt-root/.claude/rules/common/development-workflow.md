# 開発ワークフロー

> このファイルは[common/git-workflow.md](./git-workflow.md)を拡張し、Git操作の前に行われる機能開発プロセス全体について記述しています。

機能実装ワークフローは、調査、計画、TDD、コードレビュー、そしてGitへのコミットという開発パイプラインを説明します。

## 機能実装ワークフロー

0. **調査と再利用** _（新しい実装の前に必須）_
   - **まずGitHubコード検索：** 新しいコードを書く前に、`gh search repos`や`gh search code`を実行して既存の実装、テンプレート、パターンを探します。
   - **次にライブラリドキュメント：** 実装前にContext7または公式ベンダードキュメントを使用してAPIの動作、パッケージの使用方法、バージョン固有の詳細を確認します。
   - **Exaは上記2つで不十分な場合のみ：** GitHubの検索と公式ドキュメントの後に、より広範なWeb調査や発見のためにExaを使用します。
   - **パッケージレジストリを確認：** ユーティリティコードを書く前にnpm、PyPI、crates.io、その他のレジストリを検索します。手作りのソリューションよりも実績のあるライブラリを優先してください。
   - **適応可能な実装を探す：** 問題の80%以上を解決し、フォーク、移植、またはラップできるオープンソースプロジェクトを探します。
   - 要件を満たす場合は、完全に新しいコードを書くよりも、実績のあるアプローチを採用または移植することを優先してください。

1. **まず計画する**
   - **planner**エージェントを使用して実装計画を作成
   - コーディング前に計画ドキュメントを生成：PRD、アーキテクチャ、system_design、tech_doc、task_list
   - 依存関係とリスクを特定
   - フェーズに分割

2. **TDDアプローチ**
   - **tdd-guide**エージェントを使用
   - まずテストを書く（RED）
   - テストをパスするよう実装する（GREEN）
   - リファクタリング（IMPROVE）
   - カバレッジ80%以上を確認

3. **コードレビュー**
   - コード作成直後に**code-reviewer**エージェントを使用
   - CRITICALおよびHIGHの問題に対処
   - 可能な場合はMEDIUMの問題も修正

4. **コミットとプッシュ**
   - 詳細なコミットメッセージ
   - コンベンショナルコミット形式に従う
   - コミットメッセージの形式とPRプロセスについては[git-workflow.md](./git-workflow.md)を参照
